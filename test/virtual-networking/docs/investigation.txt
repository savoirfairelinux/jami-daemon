Existing internal seams that matter most

These are more important than any external framework:

  - The daemon exposes connectivityChanged() and fans it out to accounts and the global UPnP context.
  - JamiAccount::connectivityChanged() triggers conversation handling, dht_->connectivityChanged(), and connectionManager_->connectivityChanged().
  - dhtnet::UPnPContext::connectivityChanged() debounces the event and restarts IGD discovery if the host address changed or no valid IGD remains.
  - dhtnet::ConnectionManager::connectivityChanged() sends beacons on active sockets, which is directly relevant to path-change detection and recovery.
  - opendht::DhtRunner::connectivityChanged() and Dht::connectivityChanged() clear public-address assumptions, refresh routing state, and restart bootstrap
behavior when needed.

That means your lab should test both network events themselves and the daemon’s response when told connectivity changed.

2. How I’d organize this long term

The main design idea is: model access-network transitions, not radio technologies.

For the daemon, “wifi”, “mobile”, and “satellite” are mostly visible as combinations of:

  - local interface/subnet change
  - default route change
  - router/IGD change
  - public IP change
  - inbound reachability change
  - latency / jitter / loss / bandwidth change
  - allowed services changing: UPnP yes/no, multicast yes/no, STUN/TURN yes/no

That abstraction scales much better than trying to literally emulate Wi-Fi or LTE.

Suggested repository structure

Keep this under daemon/test/virtual-networking/, because the current lab already lives there and this area is not wired into the default test/meson.build
suite anyway. That’s actually good: these should stay opt-in privileged integration tests at first.

Languages

I’d use:

  - Bash for low-level networking primitives: ip, tc, nft, miniupnpd, namespaces.
  - Python for orchestration, timelines, assertions, retries, and artifact collection.
  - YAML or JSON for network profiles and scenario definitions.

I would not try to write the whole thing in Bash once scenarios become dynamic.

Separation of concerns

Keep these separate:

  - Topology: what namespaces and links exist.
  - Profile: what kind of access network a link behaves like.
  - Scenario: timeline of events.
  - Actor: what runs in each namespace.
  - Probe / assertion: how pass/fail is measured.

That separation is what will let you grow from UPnP into TURN, DHT proxy, dual-stack, roaming, captive-ish networks, and so on.

My recommendation is very simple:

  - Phase 1: keep the current architecture and grow it.
  - Phase 2: add orchestration and scenario definitions.
  - Phase 3: only then decide whether Mininet/Containernet/CORE buys you enough.

Jumping to ns-3 now would likely slow you down more than it helps.

4. First steps I’d take

First, define a scenario taxonomy

I’d define scenarios by app-visible network events:

  - local IP/subnet change
  - default-route change
  - IGD appears/disappears
  - public IP changes
  - inbound UDP reachable/unreachable
  - path profile changes
  - brief outage / flap
  - service availability changes: UPnP, STUN, TURN, bootstrap, DHT proxy

Second, preserve the current script and refactor around it

Keep setup-fake-upnp-network.sh working exactly as-is, but split its logic into reusable primitives. It is a good baseline scenario.

Third, build one new topology: dual access

This is the most important next move.

Instead of mutating one static LAN too aggressively, create a topology where the daemon namespace has two possible uplinks, for example:

  - wifi0 attached to router A / IGD A
  - cell0 attached to router B / no IGD or worse profile

Then handover becomes:

  - change route metrics
  - bring one link down / up
  - renumber interface
  - switch NAT/public IP

That is much cleaner than destroying and recreating the world mid-test.

Fourth, implement these first three scenarios

  1. Same-router renumberingThe daemon stays behind the same router/IGD, but its LAN address changes.This validates address-change handling without also
introducing a new IGD.
  2. WiFi A -> WiFi BNew subnet, new router, new IGD, new public IP.This is the cleanest “roaming between Wi-Fi networks” scenario.
  3. WiFi -> mobile/satellite profileNew route/public IP and much worse path profile.For mobile, use moderate RTT/jitter/loss and maybe no UPnP.For
satellite, use high RTT, lower bandwidth, burstier loss, maybe MTU quirks.

Fifth, add assertions before adding more scenarios

For each scenario, capture at least:

  - did the daemon notice the connectivity change
  - did UPnP mappings disappear/reappear correctly
  - is the DHT port reachable from outside again
  - did bootstrap/discovery recover
  - did active peer connections survive or reconnect
  - how long recovery took

Sixth, keep these tests opt-in at first

Because they need root / capabilities and are environment-sensitive, I would not put them into the default unit-test path yet. Later, put them in a
dedicated privileged CI job or self-hosted runner.

My concrete recommendation

If I had to choose one direction for you: turn test/virtual-networking/ into an access-network scenario harness using Bash for plumbing and Python for
orchestration.

That gives you the best ratio of effort to real coverage, and it aligns well with the actual recovery seams already present in the daemon, dhtnet, and
opendht.
